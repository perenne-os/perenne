---
id: KB-0004
title: "git can't verify its own SSH-signed commits: gpg.ssh.allowedSignersFile not configured"
status: fixed
severity: low
component: dev-environment
symptoms:
  - "git log --show-signature prints: 'error: gpg.ssh.allowedSignersFile needs to be configured and exist for ssh signature verification'"
  - "Signature shows as 'No signature' and '%G?' returns 'N', even though the commit object DOES contain a 'gpgsig -----BEGIN SSH SIGNATURE-----' header"
  - "Commits are being signed (commit.gpgsign=true, user.signingkey set) but cannot be verified locally"
diagnosis: >
  SSH signature verification needs an allowed-signers file that maps a principal
  (the committer's email) to the public key(s) trusted for that identity. Without
  gpg.ssh.allowedSignersFile pointing at such a file, git has nothing to check the
  signature against and reports it as absent/unverifiable. This is a
  verification-only gap: signing itself (KB-0001) is unaffected — the gpgsig
  header is present, it just can't be validated.
playbook:
  - "Create an allowed-signers file, e.g. C:/Users/<you>/.ssh/allowed_signers, with one line per identity: '<committer-email> <contents of your .pub key>' (e.g. 'you@example.com ssh-ed25519 AAAA... comment')."
  - "Point git at it globally: 'git config --global gpg.ssh.allowedSignersFile \"C:/Users/<you>/.ssh/allowed_signers\"'."
  - "The principal in the left column MUST match the commit's committer email, or verification still fails."
  - "Reverse if needed: 'git config --global --unset gpg.ssh.allowedSignersFile' (and delete the file)."
verification: "'git log --show-signature -1' prints 'Good \"git\" signature for <email> ...' and '%G?' returns 'G'."

## Notes

This completes the SSH-on-Windows trilogy from Phase 0:

- **KB-0001** — making git *sign* commits (redirect to system ssh-keygen).
- **KB-0002** — making git *push* over SSH (redirect to system ssh).
- **KB-0004** — making git *verify* those signatures (allowed-signers file).

The recurring lesson: signing, transport, and verification are three separate
concerns. A commit can be correctly signed (gpgsig header present) and still
report "No signature" simply because the local verifier has no trust anchor to
check it against. The fix adds trust; it does not change what was signed.
