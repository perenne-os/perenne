id: KB-0003
title: "Rust build fails on Windows with LNK1181: cannot open input file 'kernel32.lib'"
status: fixed
severity: medium
component: dev-environment
symptoms:
  - "cargo build / cargo test fails while linking with: 'LINK : fatal error LNK1181: cannot open input file kernel32.lib'"
  - "The MSVC compiler (cl.exe / link.exe) is installed, but linking host (test) executables still fails"
  - "Error appears specifically when building executables (e.g. unit tests), not just libraries"
diagnosis: >
  Rust's default Windows toolchain (the msvc target) links with Microsoft's
  link.exe, which resolves system imports such as kernel32.lib from the
  Windows SDK. Installing only the MSVC compiler (the C++ build tools) without
  the Windows SDK component leaves link.exe unable to find kernel32.lib, so any
  link step that produces an executable fails with LNK1181.
playbook:
  - "Install the Windows SDK via the Visual Studio Installer (or Build Tools for Visual Studio): under 'Desktop development with C++', enable the 'Windows 10/11 SDK' component."
  - "Open a fresh shell so the updated environment (library paths) is picked up."
  - "Re-run the build: 'cargo build; cargo test' (or './tools/build.ps1')."
  - "Reverse if needed: uninstall the Windows SDK component from the Visual Studio Installer. (No git/config state is changed by this fix.)"
verification: "cargo build and cargo test link successfully and produce/run test executables (no LNK1181). './tools/build.ps1' prints 'Build + tests OK.'"

## Notes

This was actually the *first* environment problem hit during Phase 0
(chronologically before the SSH issues, KB-0001/KB-0002), but it was given a
knowledge-base entry later — the learning note originally cited it incorrectly
against the schema example file. The lesson is mundane but important: linking is
a separate concern from compiling. A working C++ *compiler* is not enough; the
MSVC linker also needs the SDK import libraries. The same class of failure shows
up for anything that shells out to link.exe on Windows.
