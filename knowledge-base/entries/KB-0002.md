title: "git push over SSH fails with 'Permission denied (publickey)' on Windows despite a registered key"
status: fixed

symptoms:
  - "git push fails: 'git@github.com: Permission denied (publickey). fatal: Could not read from remote repository.'"
  - "The SSH key IS registered on GitHub (auth) and works when tested with the system ssh: 'ssh -T git@github.com' prints 'Hi <user>! You've successfully authenticated'"
  - "The key is loaded in the Windows ssh-agent (ssh-add -l lists it)"
diagnosis: >
  Same root cause as KB-0001. Git for Windows uses its own bundled ssh client
  (under the Git installation), not the system OpenSSH on PATH. The bundled ssh
  cannot reach the Windows OpenSSH ssh-agent (a named pipe), so it never presents
  the agent-cached key and GitHub rejects the connection. The system ssh can
  reach the agent, which is why 'ssh -T git@github.com' succeeds while 'git push'
  fails.
playbook:
  - "Point git at the system OpenSSH ssh client: 'git config --global core.sshCommand \"C:/Windows/System32/OpenSSH/ssh.exe\"'."
  - "If GitHub's host key was never accepted, accept it once: 'ssh -o StrictHostKeyChecking=accept-new -T git@github.com'."
  - "Retry: 'git push -u origin main'."
  - "Reverse if needed: 'git config --global --unset core.sshCommand'."
verification: "git push succeeds and the branch is pushed/tracked. Confirm with 'git ls-remote origin' or 'git status' showing up-to-date with origin."

## Notes

Note that KB-0001 (signing) and KB-0002 (push) are two symptoms of one underlying
cause: on Windows, git's bundled OpenSSH tooling does not integrate with the
Windows ssh-agent. The general remedy is to redirect git to the **system**
OpenSSH for both operations:

- `gpg.ssh.program` → `C:/Windows/System32/OpenSSH/ssh-keygen.exe` (signing)
- `core.sshCommand`  → `C:/Windows/System32/OpenSSH/ssh.exe` (transport)

A future self-healing rule could recognize either symptom and propose the
matching redirect, since both preserve security (the key stays in the agent)
rather than weakening it.
