# Glossary

Plain-language definitions of terms used throughout these docs. Aimed at someone new to systems programming.

- **Kernel** ‚Äî the core program of an operating system. It controls the CPU, memory, and devices, and decides which programs run when.
- **Kernel space** ‚Äî the most privileged CPU mode. Code here can access any memory and any hardware directly. A bug here can compromise the whole machine.
- **User space** ‚Äî the restricted mode ordinary programs run in. They must ask the kernel for access to resources, so a bug here is contained.
- **Monolithic kernel** ‚Äî a design where most of the OS (drivers, filesystems, networking) runs together in kernel space. Fast, but a large attack surface (e.g. Linux).
- **Microkernel** ‚Äî a design where only a tiny core runs in kernel space; everything else runs isolated in user space. Smaller attack surface; what this project uses (e.g. seL4).
- **Trusted Computing Base (TCB)** ‚Äî all the code that must be correct for the system to be secure (essentially, everything privileged). Smaller is better.
- **Capability** ‚Äî an unforgeable token that names a resource *and* grants permission to use it. A component can do only what its capabilities allow.
- **IPC (Inter-Process Communication)** ‚Äî how isolated programs talk to each other, typically by passing messages. The microkernel's main job is to deliver these safely.
- **call/reply (RPC)** ‚Äî request/response IPC: a client `call`s a server (sending a request and blocking for the answer) and the server `reply`s. The kernel records which task called so the reply returns to exactly that caller. Built on the one-way send/recv rendezvous.
- **reply capability (one-shot)** ‚Äî a capability the kernel mints when a server receives a `call`, naming the caller to answer. The server replies through it (in any order, even after receiving other calls) and the cap is consumed ‚Äî letting a server hold several calls in flight at once.
- **Post-quantum cryptography (PQC)** ‚Äî cryptography designed to resist an attacker with a large-scale quantum computer (which would break RSA/ECC). The project's baseline per ADR 0004; ordinary code on ordinary chips, no quantum hardware needed.
- **KEM (key-encapsulation mechanism)** ‚Äî a public-key scheme for establishing a shared secret: *encapsulate* to someone's public key yields a (ciphertext, secret) pair; the holder of the private key *decapsulates* the ciphertext to recover the same secret. The post-quantum replacement for classical key exchange.
- **ML-KEM** ‚Äî the NIST-standardized post-quantum KEM (FIPS 203, formerly "Kyber"). This kernel uses **ML-KEM-768** (NIST security level 3).
- **Shared secret** ‚Äî the symmetric key two parties derive via a KEM, used to key further (symmetric) encryption.
- **CSPRNG / DRBG** ‚Äî a cryptographically secure pseudo-random generator: from a small random *seed* it produces an unbounded stream that is computationally indistinguishable from random. The kernel's entropy pool is a CSPRNG built on the ChaCha20 stream cipher.
- **Entropy pool** ‚Äî the kernel's randomness source: a CSPRNG seeded by a hardware entropy device (here the virtio-rng component) that serves randomness on demand and is periodically *reseeded* with fresh device entropy (mixed into, not replacing, its state).
- **getrandom** ‚Äî the syscall a U-mode component uses to draw bytes from the kernel entropy pool. Capability-gated (the caller must hold a `Randomness` capability) ‚Äî a kernel-owned generator is exposed as a syscall because the CSPRNG can't run unprivileged.
- **HAL (Hardware Abstraction Layer)** ‚Äî a uniform interface that hides hardware differences, so the kernel doesn't need to know which specific device or chip it's running on.
- **ISA (Instruction Set Architecture)** ‚Äî the "language" a CPU speaks (e.g. x86-64, ARM64, RISC-V). Code must be built for a specific ISA.
- **RISC-V** ‚Äî a modern, open, royalty-free ISA. Our first target because it's clean to learn and future-forward.
- **QEMU** ‚Äî software that emulates a whole computer, so we can run and test our kernel safely on an existing machine.
- **Emulator** ‚Äî a program that imitates real hardware, letting software run as if on the real thing.
- **`no_std`** ‚Äî a Rust mode that drops the standard library (which assumes an OS exists). Required for kernel code, since *we are* the OS.
- **Freestanding binary** ‚Äî a program that runs without an underlying operating system or runtime ‚Äî like a kernel.
- **Cross-compilation** ‚Äî building a program on one kind of machine (e.g. an x86-64 laptop) to run on a different kind (e.g. a RISC-V computer).
- **Bootloader / firmware** ‚Äî the first code that runs when a machine powers on; it initializes hardware and then hands control to the kernel.
- **OpenSBI** ‚Äî the standard firmware/runtime layer for RISC-V that starts before the kernel; QEMU includes it by default.
- **SBI (Supervisor Binary Interface)** ‚Äî the standard interface a RISC-V kernel (S-mode) uses to request services from firmware (M-mode), e.g. "print this byte". OpenSBI implements it.
- **ecall** ‚Äî the RISC-V instruction that traps into the next-more-privileged mode; how the kernel calls the SBI (and how user programs will call the kernel later).
- **hart** ‚Äî RISC-V's term for a hardware thread (a CPU core as the software sees it).
- **M-mode / S-mode / U-mode** ‚Äî RISC-V privilege levels: machine (firmware, most privileged), supervisor (the kernel), user (applications).
- **Linker script** ‚Äî instructions to the linker about where in memory each part of a binary must be placed; essential on bare metal where addresses are physical and fixed.
- **wfi (wait for interrupt)** ‚Äî RISC-V instruction that puts a hart to sleep until an interrupt arrives; our kernel's idle loop.
- **Post-quantum cryptography (PQC)** ‚Äî encryption algorithms designed to resist attacks from future quantum computers; ordinary software running on ordinary chips.
- **ADR (Architecture Decision Record)** ‚Äî a short, dated note recording one significant decision and the reasoning behind it.
- **Self-healing / knowledge organism** ‚Äî this project's support model: the OS keeps a growing memory of issues and proven fixes and consults itself to diagnose and repair problems.
- **Safety cage** ‚Äî the rule that every automated self-healing action must be capability-checked, logged, reversible, and auditable, so the healer can never gain unchecked power.
- **Playbook** ‚Äî the ordered, reversible set of fix steps recorded for a known issue; what the self-healer applies (inside the safety cage) once a problem is diagnosed.
- **Diagnosis (rule engine)** ‚Äî the deterministic, explainable matching of an observed symptom (e.g. a fault cause) to a known issue ‚Äî a transparent table lookup, never a black box. In this kernel it is `heal::diagnose`, which is why it is safe to run in the kernel itself.
- **Launch generation** ‚Äî the count of how many times a task has been (re)started, handed to the task by the kernel at launch (in `a0`). It lets a component tell a fresh start from a restart ‚Äî e.g. so a transient fault can be retried into success.
- **Accelerator (GPU / NPU / TPU / QPU)** ‚Äî a specialized coprocessor a normal CPU offloads work to (graphics, AI math, or quantum operations). Treated as a *device*, not something an OS runs on.
- **Trap** ‚Äî the CPU's "stop and handle this" mechanism: on an exceptional event the hart jumps to the kernel's registered handler. Exceptions and interrupts are the two flavors.
- **Exception** ‚Äî a synchronous trap caused by the current instruction itself (e.g. a breakpoint or an illegal instruction).
- **Interrupt** ‚Äî an asynchronous trap arriving from outside the instruction stream (e.g. the timer). The kernel handles it and resumes the interrupted code.
- **PLIC (platform-level interrupt controller)** ‚Äî the RISC-V device that collects external device interrupts and routes the highest-priority pending one to a hart context. The kernel *claims* an interrupt (learns which IRQ) and *completes* it (acknowledges) via PLIC registers.
- **IRQ (interrupt request)** ‚Äî a numbered interrupt line from a device (here virtio-rng is IRQ 8). The PLIC maps IRQ numbers to priorities and per-context enables.
- **claim / complete** ‚Äî the PLIC handshake: reading the claim register returns (and latches "in service") the pending IRQ; writing it back completes it, re-arming the source.
- **CSR (Control and Status Register)** ‚Äî special per-hart registers that configure and report CPU state, accessed with dedicated instructions (`csrr`/`csrw`). Trap handling lives in CSRs like `stvec` (handler address), `scause` (why), `sepc` (where), `stval` (extra detail).
- **Trap frame** ‚Äî the snapshot of every register saved on trap entry and restored on exit, so the interrupted code never notices. A saved frame is effectively a paused task ‚Äî the seed of context switching.
- **`sret`** ‚Äî the return-from-trap instruction: restores the pre-trap privilege mode and jumps back to `sepc`.
- **`ebreak`** ‚Äî the RISC-V breakpoint instruction; Phase 2a triggers one on purpose to prove trap recovery works.
- **Timebase** ‚Äî the fixed rate the `time` counter ticks at (10 MHz on QEMU virt), independent of CPU clock speed; deadlines for timer interrupts are expressed in these ticks.
- **Device tree / FDT (flattened device tree)** ‚Äî a firmware-provided data structure describing the hardware (CPUs, memory, devices) that a portable kernel reads at boot instead of hardcoding values. The *flattened* form is a compact big-endian binary (header + a node/property token stream + a strings block); OpenSBI passes its address to the kernel in a register. Phase 4a parses it.
- **`timebase-frequency`** ‚Äî the device-tree property giving the rate of the `time` counter; the kernel reads it to derive timer deadlines (replacing the hardcoded timebase).
- **Memory node** ‚Äî the device-tree node whose `reg` property gives the RAM base address and size; the kernel reads it to learn how much memory it has.
- **MMIO (memory-mapped I/O)** ‚Äî device registers exposed at fixed physical addresses and accessed with ordinary loads/stores; in S-mode the kernel must map those pages into its page table (and, here, into every address space) before touching them.
- **virtio** ‚Äî a standard interface for *paravirtualized* devices (rng, block, net‚Ä¶): the guest talks to a simple, well-specified device the hypervisor provides, instead of emulating real hardware. Our entropy driver uses virtio-mmio (the memory-mapped transport).
- **Virtqueue** ‚Äî the shared-memory channel between a virtio driver and device: a descriptor table (buffers) plus an *available* ring (driver‚Üídevice) and a *used* ring (device‚Üídriver). The driver publishes a buffer and the device returns it filled.
- **DMA (direct memory access)** ‚Äî a device reading/writing system memory directly, at *physical* addresses, without the CPU copying each byte. The virtqueue and its buffers live in DMA memory shared with the device.
- **Block device** ‚Äî a storage device addressed in fixed-size *sectors* (here 512 bytes) rather than as a byte stream; you read or write whole sectors by number. The disk is a block device (virtio-blk).
- **virtio-blk** ‚Äî the virtio block-device interface. Each request is a 3-descriptor chain (a header naming the operation + sector, a data buffer, and a status byte) submitted through the virtqueue.
- **Filesystem** ‚Äî the layer that turns named files into block reads/writes over a block device. Phase 6b's is minimal and read-only: it locates a file by name and reads its bytes.
- **Superblock** ‚Äî a filesystem's first block: format magic/version, the block size, and where the directory lives. The kernel reads it to find everything else.
- **Directory entry (dirent)** ‚Äî a fixed-size record mapping a file name to its on-disk location (start block + byte length). Phase 6b packs these in one directory block.
- **Extent** ‚Äî a contiguous run of blocks holding a file's data; in Phase 6b's format a file is a single extent (start block + length), so reading it is a sequential block loop.
- **Block cache** ‚Äî a small in-memory hold of recently-read disk blocks, so a filesystem can re-read a block without a fresh device request. Phase 6b's is a single block (the one resident in the shared DMA page) ‚Äî the abstraction boundary between the filesystem and the block driver.
- **UART** ‚Äî the serial-port controller that turns bytes into the serial line signal; the kernel's text console.
- **ns16550** ‚Äî the ubiquitous 16550-family UART (QEMU virt and many boards), programmed via a few byte registers spaced by `reg-shift`.
- **`THR` / `LSR` / `THRE`** ‚Äî the 16550 transmit-holding register (where you write a byte), the line-status register, and its "holding register empty" bit (`0x20`) the driver polls before transmitting.
- **User-space component / server** ‚Äî an unprivileged task that provides a service or drives a device, reached over capability-checked IPC; the unit of extension per [ADR 0007](decisions/0007-extensibility-user-space-components.md). Adding one neither grows nor endangers the trusted core.
- **Device driver (as a component)** ‚Äî such a component whose authority is a device's MMIO mapped into its address space (and nowhere else) plus an IPC endpoint ‚Äî so isolation and capabilities bound exactly what it can touch and who can call it. The first one is the RTC driver.
- **RTC (real-time clock)** ‚Äî a hardware clock giving wall-clock time; here the goldfish RTC, owned by the first user-space driver component.
- **Physical address** ‚Äî a real location in RAM chips. Before Phase 2b, the only kind of address the kernel had.
- **Virtual address** ‚Äî the address code actually uses once the MMU is on; translated to a physical address through page tables on every access.
- **Page / frame** ‚Äî the same 4 KiB unit seen from two sides: a *page* is virtual, a *frame* is the physical RAM behind it.
- **MMU (Memory Management Unit)** ‚Äî CPU hardware that translates virtual to physical addresses and enforces per-page permissions, faulting on violations.
- **Page table** ‚Äî the tree the MMU walks to translate addresses. In Sv39 it has three levels; each table is one 4 KiB frame of 512 entries.
- **PTE (Page Table Entry)** ‚Äî one slot in a page table: the next level's (or final frame's) number plus permission flags (Valid/Read/Write/eXecute‚Ä¶).
- **Sv39** ‚Äî RISC-V's three-level paging scheme: 39-bit virtual addresses, 4 KiB pages. What Phase 2b enables.
- **`satp`** ‚Äî the CSR that holds the root page table's frame number and translation mode; writing it turns paging on (Phase 2b) and, from Phase 3b-ii, selects the running task's address space ‚Äî the scheduler writes it on every context switch.
- **Address space** ‚Äî the set of virtual‚Üíphysical mappings a task sees, named by its `satp`. From Phase 3b-ii each U-mode task has its own, so one task can't reach another's memory.
- **Global mapping (`PTE_G`)** ‚Äî a PTE bit marking a translation as the same in every address space (the TLB needn't flush it across an `satp` change). It is only a TLB *hint*: the mapping must still physically exist in each tree being walked ‚Äî so Phase 3b-ii *clones* the kernel into every per-task page table rather than relying on `G` alone.
- **TLB (Translation Lookaside Buffer)** ‚Äî the MMU's cache of recent translations; must be flushed (`sfence.vma`) when mappings change.
- **`sfence.vma`** ‚Äî the RISC-V instruction that flushes the TLB so stale translations can't survive a page-table change.
- **Identity mapping** ‚Äî mapping each virtual address to the identical physical address. Changes nothing about *where* things are ‚Äî and everything about *what's allowed*, via permission bits.
- **W^X (write XOR execute)** ‚Äî the policy that no memory is both writable and executable: code can't be overwritten, data can't be run. Phase 2b's payoff.
- **Guard page** ‚Äî a deliberately unmapped page (here: below the stack) so an overflow faults immediately instead of corrupting the neighbor.
- **Frame allocator** ‚Äî the kernel component that owns physical RAM and hands out/reclaims 4 KiB frames; ours tracks them in a bitmap.
- **Task / kernel thread** ‚Äî an independently schedulable unit of execution inside the kernel: a stack plus a saved register context. Phase 2c runs three of them.
- **Context switch** ‚Äî saving the registers of the running task and restoring another's, so the CPU resumes a different task as if it had never stopped.
- **Context (callee-saved frame)** ‚Äî the minimal register set a *voluntary* switch must preserve: `ra`, `sp`, and `s0..s11`. Smaller than a trap frame because the calling convention already saved the rest.
- **Run queue** ‚Äî the kernel's list of runnable tasks and the bookkeeping that picks which runs next; ours is a fixed array scanned round-robin.
- **Round-robin** ‚Äî the simplest fair scheduling policy: give each ready task a turn in rotation, then start over.
- **Cooperative scheduling** ‚Äî tasks keep the CPU until they voluntarily `yield`. Simple, but one selfish task can starve the rest.
- **Preemptive scheduling** ‚Äî the kernel forcibly takes the CPU back (here, on a timer tick), so no task can monopolize it.
- **Yield** ‚Äî a task voluntarily giving up the CPU to let another run; in this kernel, `yield_now()`.
- **Time slice / tick-policy hook** ‚Äî the decision made on each timer tick about whether to preempt the running task; the point where the heartbeat becomes a scheduler clock.
- **Trampoline** ‚Äî a tiny stub a new task first lands in: it sets up the task's running conditions (here, enabling interrupts) before jumping to the real entry function.
- **System call (syscall)** ‚Äî the controlled entry by which an unprivileged user task requests a service from the kernel; in this kernel, an `ecall` from U-mode decoded by number (`print`, `exit`).
- **Syscall ABI** ‚Äî the register convention for a syscall: here `a7` selects the call, `a0..` pass arguments, and `a0` returns the result.
- **`sret`** ‚Äî the RISC-V instruction that returns from a trap to a lower or equal privilege; reading `sstatus.SPP` decides whether it lands in U-mode or S-mode.
- **`sstatus.SPP`** ‚Äî the "supervisor previous privilege" bit: records whether a trap came from U-mode (0) or S-mode (1), and selects where `sret` returns.
- **`sstatus.SPIE` / `SIE`** ‚Äî the saved and live supervisor interrupt-enable bits; `sret` restores `SIE` from `SPIE`, so a task entered with `SPIE = 1` resumes with interrupts on.
- **`sscratch`** ‚Äî a scratch CSR used as a privilege-aware stack pointer: it holds the kernel trap-stack top while a user task runs and 0 while the kernel runs, letting the trap entry swap to a trusted stack only when needed.
- **Trap stack / kernel stack** ‚Äî the trusted stack a trap from U-mode must switch to before the kernel touches it, since the interrupted user stack is untrusted.
- **`sstatus.SUM`** ‚Äî "permit Supervisor User Memory access": when 0 (the default), S-mode accesses to user pages fault; the `print` syscall opens it briefly for a validated copy.
- **Confused deputy** ‚Äî a privileged component tricked into misusing its authority on behalf of a less-privileged caller; here, the kernel reading a kernel address supplied as a user `print` pointer ‚Äî prevented by validating the pointer against the user's own memory.
- **Task termination / containment** ‚Äî ending a task cleanly (`exit`) or because it faulted (killed); a U-mode fault contains just the task, leaving the kernel running.
- **U bit (PTE)** ‚Äî the page-table entry flag that makes a page accessible to U-mode; kernel pages omit it, so a user touch faults.
- **Capability table (CSpace)** ‚Äî a task's private array of capabilities; a syscall names a kernel object by an *index* into this table, which is what makes the token unforgeable (you can only name what you were granted). In this kernel, `Task.caps`.
- **Endpoint** ‚Äî a synchronous IPC rendezvous point named by a capability; `send`/`recv` meet here. In Phase 3b-iii it is a symbolic id whose wait queue is the set of tasks blocked on it.
- **Synchronous IPC / rendezvous** ‚Äî message passing where sender and receiver must meet: whichever arrives first blocks until the other does, then the message transfers and both proceed.
- **Blocking / wait queue** ‚Äî a task that cannot proceed (e.g. `recv` with no sender) is parked in a non-runnable `Blocked` state and skipped by the scheduler until a peer wakes it; the "queue" is the set of tasks blocked on a given endpoint.
- **`yield` syscall** ‚Äî a U-mode task cooperatively gives up the CPU to the next ready task through the kernel (it cannot call the scheduler directly). Syscall number 3 in this kernel.
- **task `Exited` state** ‚Äî a terminated task slot the scheduler skips forever; its stacks are not yet reclaimed (reaping is deferred).
- **`user_trampoline`** ‚Äî the first-run entry that `sret`s a freshly spawned U-mode task into U-mode (sets `sscratch`/`sepc`/`sstatus`, switches to the user stack); the U-mode analogue of `task_trampoline`.
